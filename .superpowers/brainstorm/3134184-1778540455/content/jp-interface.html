What's JP's interface to the company system?

JP has 10 h/wk (then 5), Spanish-speaking, uses AI heavily but no formal technical training, lives in another timezone. He needs to read, review, propose prompt changes, and act on notifications — but he is NOT the operator. Three patterns, very different blast radii.

A — HEAVY
JP's laptop:
• Claude Code installed
• Gemini CLI installed
• Full git clone of repo
• Own .env (Anthropic + tokens)
• SSH key to Gitea
• Vault key (gocryptfs)

Workflow:
Opens Claude Code at any folder. Runs
full sessions like Ricardo. Edits any file.

Secrets live: on JP's laptop
Bus factor: 0 (independent)
Onboarding: hours, plus ongoing maintenance

A. Heavy — JP is a peer operator

Same powers as Ricardo. Highest autonomy. Heaviest install + ongoing tooling burden on JP. Doubles the surface where credentials live.

B — HOSTED (recommended)
JP's laptop:
• Web browser only
• Tailscale client (or public TLS)

On Ricardo's server:
• Per-user code-server (web IDE)
• Per-user Claude Code session
• Shared repo, separate workdirs
• Secrets sit on the server

Workflow:
JP logs into code-server in browser.
Same Claude Code experience, no install.

Secrets live: on Ricardo's server only
Bus factor: 1 (depends on server uptime)
Onboarding: minutes (browser login)

B. Hosted — JP works through a browser into the server

Same power as A, but the tools run on Ricardo's hardware. No JP install, no JP-side secrets. Requires server uptime + a second Anthropic license (or per-session login). New piece to operate: code-server.

C — LIGHT
JP's laptop:
• Web browser
• Telegram app
• Vault key only (for sensitive docs)

JP's reach:
• Gitea web UI (read + edit markdown)
• Telegram bot (chat with system)
• Download/upload .docx via Gitea

Workflow:
JP reads in Gitea, comments via bot, edits
markdown in browser. Ricardo's sessions
integrate JP's notes.

Secrets live: on Ricardo's server only
Bus factor: 1 (depends on Ricardo)
Onboarding: zero — JP already has both apps

C. Light — JP only uses Gitea web + Telegram bot

Lowest friction for JP, highest reliance on Ricardo as operator. JP can't independently run agents. Telegram bot becomes JP's main system surface — fits his actual role (strategy + review) better than A.

My recommendation: start with C, design forward-compatibly so we can layer B on later if JP wants more autonomy. C matches JP's actual contribution (strategy + review + occasional prompt input) without forcing a tooling commitment he hasn't signed up for. If we build C well, JP will likely never need B.
